Intro/Getting started

Global Infra

  • Data at some AWS data center
  • Cluster of these ⇒ Availability Zone
  • Cluster of AZs ⇒ Region
  • Regions like London, N. Virgina, etc
  • Provides resilience to hardware failures, natural calamities, etc
  • All of these connected by low latency links

Choice of Region

  • Compliance: data should stay in UK only
  • Latency: data should be close to users geo location
  • Pricing: diff prices for diff countries due to tax norms
  • Service Availability: some services aren't available in some regions

Interacting with AWS API

  • GUI: aka AWS management console
  • CLI: through terminal
  • SDK: through general purpose programming languages, java, py, js, etc

IAM Users

  • Email signed up with has unrestricted access to everything, is the root user. It has all of the powers that can be had.
  • Create IAM users for all individual users, and create groups for define accesses
  • Also there's authentication and authorisation, to call any services service, even if they belong to the same aws account(say to store image from EC2 to S3) auth is required, this is on aws level only (irrespective of whether or not the services are owned by same of different account), application level auth stuff is to e implemented by the customer.
  • In short, most aws api calls must be signed and authenticated

IAM Roles

  • So there's 2 things, IAM User & IAM Roles: both are identities, User is typically used by individual users, while roles are used by "someone" or "something", say your code to call another aws service.
  • This way you don't need to create IAM User for the code, nor do you need to hardcode any credentials.

IAM Users

  • Have login(static) credentials and credentials

IAM Roles

  • No login creds (username&password) Only crews to sign requests
  • Assigned programmatically
  • Temporary crews for configurable amount of time
  • Creds expire and are rotated

How it works?

  1. Admin creates an IAM Role that grants access to S3 bucket.
  1. Launch EC2 instance with IAM Role.
  1. App retrieves IAM Role credentials from EC2 instance.
  1. App makes API calls using temporary credentials.

Steps:

Console >> IAM >> Roles >> Create a Role >> Select trusted entity to assume the role(EC2 in our case) >> Choose policy (AWS managed / custom) >> add tags >> name, description >> done

  • pick s3fullaccess for dynamoDB for the project.
  • policy JSON can be viewed by expanding.
  • If Identity Provider already exists, it can be used to create "federated" users, AWS SSO can make this easier.

Compute

Intro

  • Services like EC2, ECS(container), EKS(Kubernetes), etc.
  • We can't launch EC2 instance into space, it needs to go somewhere.
  • In AWS, each of this goes in a network (aka VPC).
  • We need to configure this VPC ourselves, a lot of default values are provided though.

Compute as a Service

  • Doing anything, basically running software needs compute resources.
  • Having on-premises servers, take time & money to buy and get delivered, time to setup, rent/space to keep, power to run, efforts to backup, secure, update, repair and maintain, and you can't get your money & time back when you don't need them, you're stuck with it, you use it or not.
  • CaaS model comes to the rescue here.
  • 3 types of compute services: VM, container, server less.

EC2

EC2 Intro

  • EC2 is compute service, most flexible one available
  • to spin it up, need to configure h/w and os
  • To choose OS, pick a AMI (Amazon Machine Image) from the ones provided by aws or community ⇒ no installation drives, wizards, etc, basically custom ISOs
  • While choosing, there are 2 components:
    • instance type: tells the blend of h/w capabilities
    • instance size: nano...xxxlarge
    • ex. t3.medium, or a1.large

EC2 Life Cycle

Container Services

  • 2 of them: ECS & EKS ⇒ container orchestration services
  • We need processes for starting, stopping, restarting, monitoring, etc the containers running across not just 1 EC2 instance, but a number of them (aka EC2 cluster)
  • This process is called Container Orchestration ⇒ hard to do on your own and doing it manually isn't scalable (as you approach thousands of containers)
  • ECS & EKS are orchestration tools help with this, just a little different ways offering a bit different features
  • These are just container orchestration (or management) services, you need to use some compute service separately (i.e. EC2) and you need ECS agent installed on your EC2 instance.

Serverless

  • You are responsible for patching, updates, highly available architecture, gives you a lot of control over the environment, but sometimes its not required. Sometimes, you want to focus on the business solution and not do the other stuff while still getting the advantages of the other stuff. This is serverless.
  • You can't see or access the underlying infra/instances hosting your solution, all the scaling, fault tolerance, etc is taken care of, so you focus only on the application, don't need no operational processes.
  • AWS Fargate is a serverless compute alternative to EC2 to run containers on it.
  • It scales on demand, and fault tolerance is built-in.
  • Trade-off: Control vs Convinience.

AWS Lambda

  • You write some code, package it, upload it, it's converted into a lambda function.
  • These don't keep running, they're triggered.
  • When triggered, AWS runs it in an isolated environment, managed by AWS.
  • Suitable for stuff like backend processing, or report generation, or resizing thumbnails, you don't need this stuff to run 24x7.
  • Currently the limit it 25 minutes, and you're billed to the nearest 100ms.
  • Both options of EC2 and serverless available.

Networking

VPCs

  • virtual private cloud: the network in which our applications live.
  • depending on requirements, default/custom VPC maybe used.
  • Nothing goes in/out of VPC without the owner's explicit permission.

Creating VPC & subnets

  • Creating VPC: tell the Region & IP Range (CIDR).
  • Creating Subnets: tell VPC, AZ, IP Range.
  • Subnets provide us "granular" control over these networks.
  • Ex. the public data could go on the internet enabled subnet, while private stuff will be hosted in the private network.
  • To achieve this: there's Internet Gateway IGW & Virtual Private Gateway VGW for internal corporate network(establishes an ecnrypted VPN) .

VPC Routing

  • We call the subnets public/private but its really the routing tables that decide this.
  • If there exists an entry in the route table of the internet gateway to the subnet, it means it has internet/public access.

Security

  • 2 ways its provided in AWS: n/w ACLs (n/w access control) & Security groups.
  • n/w ACL is like a firewall for each of the subnets.
  • Security groups are like firewall on the EC2 level.
  • for ACLs everything is allowed by default, you can block stuff by adding rules. for Sec Groups everything is blocked by default, you can allow stuff by adding rules.
